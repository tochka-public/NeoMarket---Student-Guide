# NeoMarket — Брифинг к Protocol Summit

Читать до встречи. Здесь — то, что удалось восстановить из системы NeoMarket. Часть данных побита, часть потеряна. На Protocol Summit будем восстанавливать недостающее вместе.

---

## Что такое NeoMarket

Маркетплейс товаров. Три роли: **продавцы** (создают товары), **модераторы** (проверяют перед публикацией), **покупатели** (ищут и покупают).

Три независимых сервиса, у каждого своя БД. Общаются по HTTP и через события.

---

## Общая схема

```
┌──────────────────┐                         ┌──────────────────┐
│                  │   события:               │                  │
│   B2B            │   ???                    │   Moderation     │
│   (продавец)     │ ──────────────────────►  │   (модератор)    │
│                  │                          │                  │
│   Создание       │   события:               │   Очередь        │
│   товаров,       │   ???                    │   проверки,      │
│   SKU,           │ ◄──────────────────────  │   одобрение,     │
│   накладные      │                          │   блокировка     │
│                  │                          │                  │
└────────┬─────────┘                          └──────────────────┘
         │
         │  HTTP API (синхронно)
         │  ???
         │
         ▼
┌──────────────────┐
│                  │
│   B2C            │
│   (покупатель)   │
│                  │
│   ???            │
│                  │
└──────────────────┘
```

**Правило**: каждый модуль — отдельный сервис со своей БД. Один сервис не лезет в базу другого. Всё — только через API.

**Что есть**: общая структура, основные сущности, часть эндпоинтов.
**Чего нет**: формат событий между сервисами, куча деталей API, весь модуль B2C.

---

## Фрагмент 1: B2B — Кабинет продавца

Продавец создаёт товары, добавляет варианты (SKU — это конкретная конфигурация: размер, цвет, объём памяти), управляет ценами. Когда товар готов — отправляет накладную на склад.

### Сущности (восстановлены)

- **Product** — карточка товара (название, описание, категория, фото, характеристики)
- **SKU** — конкретный вариант для продажи. iPhone 256GB Black = один SKU, iPhone 256GB White = другой. У каждого SKU своя цена, артикул, фото, остаток на складе
- **Category** — дерево категорий (Электроника → Смартфоны → Android)
- **Characteristic** — свойства товара и SKU (бренд, цвет, размер, объём памяти)
- **Invoice** — накладная: список SKU с количествами для отправки на склад

### Эндпоинты (частично)

| Метод | Путь | Что делает |
|-------|------|-----------|
| POST | `/api/v1/products` | Создать товар |
| GET | `/api/v1/products/{id}` | Получить товар со всеми SKU |
| PUT | `/api/v1/products/{id}` | Изменить товар |
| POST | `/api/v1/skus` | Создать SKU |
| PUT | `/api/v1/skus` | Изменить SKU |
| POST | `/api/v1/invoices` | Создать накладную |
| POST | `/api/v1/invoices/accept` | Принять накладную (склад) |
| ??? | ??? | Что-то для B2C — связь с витриной покупателя |

### Статусы товара

```
CREATED  →  ON_MODERATION  →  MODERATED  (товар опубликован)
                               или
                            →  BLOCKED    (продавец видит причину, может исправить)
```

### Что не восстановлено

- Как B2B сообщает Moderation что товар создан или изменён? Какой формат события?
- Как B2B узнаёт что товар одобрен или заблокирован? Что приходит от Moderation?
- Как B2C получает товары из B2B? Какие эндпоинты нужны для каталога?
- Что происходит с остатками когда покупатель оформляет заказ?

## Фрагмент 2: Moderation — Проверка товаров

Модератор — сотрудник маркетплейса. Берёт карточку из очереди, смотрит, решает: одобрить или заблокировать.

### Как работает (общая логика)

1. Moderation как-то узнаёт что товар создан или изменён *(формат события — ???)*
2. Запрашивает данные товара из B2B (`GET /products/{id}`)
3. Сохраняет снимок товара. Если товар изменён — два снимка: "до" и "после"
4. Модератор берёт карточку из очереди и смотрит
5. Одобряет или блокирует — и как-то сообщает B2B *(формат события — ???)*

### Эндпоинты (частично)

| Метод | Путь | Что делает |
|-------|------|-----------|
| POST | `/api/v1/product-moderation/get-next` | Следующая карточка из очереди |
| POST | `/api/v1/products/{id}/approve` | Одобрить |
| POST | `/api/v1/products/{id}/decline` | Заблокировать (с причиной) |
| GET | `/api/v1/product-blocking-reasons` | Список причин блокировки |

### Пример: блокировка

Модератор видит: на фото Samsung, а в описании написано iPhone. Что отправить в decline?

```
POST /api/v1/products/42/decline

{
    ???
}
```

Подумайте: что нужно, чтобы продавец понял что исправить? Достаточно общей причины — или надо указать конкретные поля?

### Что не восстановлено

- Формат события B2B → Moderation (что именно B2B шлёт когда товар создан?)
- Формат события Moderation → B2B (что Moderation шлёт после проверки?)
- Как модератор видит "что изменилось" при повторной проверке? Как хранить diff?
- Два модератора работают одновременно — как не выдать им одну карточку?
- Нужны ли приоритеты в очереди? Какие товары проверять первыми?

## Фрагмент 3: B2C — Витрина покупателя (потерян)

B2C — то, что видит покупатель. Каталог, поиск, корзина, заказы, избранное.

**Этот модуль потерян полностью.** Есть только общее представление.

### Что известно

- B2C **не хранит** товары — запрашивает из B2B по HTTP
- B2C хранит свои данные: корзина, заказы, избранное
- Покупатель видит только товары со статусом MODERATED и с остатком > 0
- Корзина **не резервирует** товар. Резервирование — только при оформлении заказа

### Домены (известны)

| Домен | Что делает |
|-------|-----------|
| **Каталог** | Список товаров с фильтрами, сортировкой, пагинацией |
| **Карточка товара** | Полная информация, выбор SKU, цена, наличие |
| **Корзина** | Добавить, удалить, изменить количество |
| **Оформление заказа** | Из корзины → резервирование → заказ |
| **Избранное** | Сохранить товар, вернуться позже |
| **Главная страница** | Баннеры, подборки товаров |

### Что нужно спроектировать (всё)

- Какие эндпоинты? Какие HTTP-методы? Какие пути?
- Какие поля в запросах и ответах?
- Как B2C запрашивает товары у B2B? Какие параметры?
- Какие статусы у заказа? Кто и когда их меняет?
- Что происходит когда товар заблокировали, а он уже в корзине у кого-то?

## Путь товара (общая логика)

```
1. Продавец создаёт товар в B2B          → статус CREATED
2. Продавец добавляет варианты (SKU)      → товар уходит на модерацию (как?)
3. Moderation получает данные товара      → модератор проверяет
4. Одобрение или блокировка              → B2B узнаёт результат (как?)
5. Продавец создаёт накладную            → склад принимает → остатки обновлены
6. Покупатель видит товар в каталоге     → B2C запрашивает из B2B (как?)
7. Покупатель добавляет в корзину        → запись в БД B2C (НЕ резервирует)
8. Покупатель оформляет заказ            → резервирование в B2B (как?)
9. Заказ создан, корзина очищена
```

Шаги 2, 4, 6, 8 — это то, что будем проектировать на Protocol Summit.

## Вопросы для размышления

1. Продавец создал товар и добавил SKU. Как Moderation об этом узнает? Что отправлять — весь товар целиком или только ID?

2. Модератор заблокировал товар. Как B2B об этом узнает? Что должно быть в сообщении, чтобы продавец понял что исправить?

3. Покупатель вчера добавил товар в корзину. Сегодня модератор его заблокировал. Покупатель открывает корзину — что он видит?

4. Два покупателя одновременно оформляют заказ на последний экземпляр. Кто получит товар?

5. В категории "Смартфоны" 200 товаров. Покупатель хочет фильтр по бренду и цене. Как передать фильтры в запросе к B2B?
