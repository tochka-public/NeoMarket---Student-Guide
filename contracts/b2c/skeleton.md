# B2C -- Скелет для Protocol Summit

Этот документ -- ваша отправная точка для проектирования API модуля B2C. Здесь описаны домены (области ответственности), но **конкретные эндпоинты вы придумываете сами**.

Для каждого домена: прочитайте описание, разберите user story, ответьте на вопрос "Что если?", и спроектируйте набор эндпоинтов.

---

## Напоминание: откуда берутся данные

| Данные | Где хранятся | Как B2C их получает |
|--------|-------------|---------------------|
| Товары, SKU, категории, характеристики | B2B (чужая БД) | HTTP API: `GET /api/v1/products/...` |
| Корзина | B2C (своя БД) | Своя таблица `cart_items` |
| Заказы | B2C (своя БД) | Свои таблицы `orders`, `order_items` |
| Избранное | B2C (своя БД) | Своя таблица `wishlist_items` |
| Подборки, баннеры | B2C (своя БД) | Свои таблицы `collections`, `banners` (Django admin) |
| Резервирование остатков | B2B (чужая БД) | HTTP API: `POST /api/v1/reserve` |

**Правило видимости**: покупатель видит товар ТОЛЬКО если `status=MODERATED`, `deleted=false`, `blocked=false` и хотя бы у одного SKU `activeQuantity > 0`.

**Идентификация пользователя**: корзина, заказы и избранное привязаны к `userId`. Механизм передачи userId (заголовок, токен, путь) — проектируется вами. Авторизация вынесена за скобки: считайте, что userId уже известен и доступен в каждом запросе.

**Категории и характеристики**: B2C не хранит справочники. Категории (дерево) и характеристики (для фильтров) запрашиваются из B2B. Подумайте, какие эндпоинты вам нужны от B2B — и запросите их у команд, делающих B2B.

## Домен 1: Каталог

### Что делает

Покупатель заходит на сайт, выбирает категорию (или вводит поисковый запрос) и видит список товаров. Может фильтровать по характеристикам (бренд, цвет, размер), по цене, сортировать. Товары отображаются постранично.

### Какие данные нужны

- Список категорий (дерево: Электроника > Смартфоны > Apple) -- из B2B
- Хлебные крошки (breadcrumbs): путь от корня до текущей категории -- строится из дерева категорий
- Список товаров в категории (с пагинацией) -- из B2B
- Характеристики для фильтров (зависят от категории) -- из B2B
- Карточки товаров в списке: название, главное фото, цена (минимальная по SKU), наличие

### User story

> Маша открывает NeoMarket, нажимает "Электроника" > "Смартфоны". Видит 48 товаров. Ставит фильтр "Бренд: Apple", "Цена: до 100 000 руб.". Остается 5 товаров. Сортирует "по цене (дешевле)". Нажимает на первый -- попадает на карточку товара.

### Что нужно спроектировать

- Как покупатель получает дерево категорий?
- **Формат фильтрованного запроса к B2B API** — конкретный формат query string:
  - Покупатель в категории "Смартфоны" хочет: бренд = Samsung, память = 256 ГБ, цена от 30 000 до 80 000. Зафиксируйте конкретную query string для этого запроса.
  - Как передать несколько значений одного фильтра: "бренд = Samsung ИЛИ Apple"? Как передать диапазон цены?
  - Как одна и та же ручка обрабатывает фильтры, которые разные для каждой категории? (У смартфонов — "память", у одежды — "размер")
- **Доступные фильтры**: как B2C узнаёт, какие фильтры доступны для категории "Смартфоны"? Нужен ли отдельный эндпоинт `GET /filters?categoryId=X`? Или фильтры приходят вместе с товарами?
- Как работает текстовый поиск?
- Какой формат ответа для списка товаров? (нужна ли пагинация в ответе?)
- Сортировка: по цене, по новизне — формат параметра?

### Что если...?

> Покупатель открыл каталог, видит 20 товаров на первой странице. Пока он листал, модератор заблокировал 3 товара с первой страницы. Покупатель нажимает "Страница 2" -- и видит дубликаты, потому что товары сдвинулись. Как с этим быть?

> Покупатель применяет фильтр "бренд = Samsung", результат — 12 товаров. Затем добавляет "память = 256 ГБ". Оказывается, что среди Samsung нет ни одного с 256 ГБ — результат пустой. Как B2C об этом узнает **заранее**, до того как покупатель нажал "применить"? Нужны ли счётчики ("Samsung (12)", "256 ГБ (0)") — и если да, откуда B2C их берёт?

## Домен 2: Карточка товара

### Что делает

Покупатель нажимает на товар и видит полную информацию: все фотографии, описание, характеристики, выбор вариации (SKU -- размер, цвет, объем), цену и наличие выбранного SKU. Отсюда можно добавить в корзину или в избранное.

### Какие данные нужны

- Полная информация о товаре (title, description, images, characteristics) -- из B2B
- Список SKU с ценами и остатками -- из B2B
- Флаг "в избранном" -- из своей БД
- Флаг "уже в корзине" (опционально) -- из своей БД

### User story

> Маша открыла "iPhone 15 Pro Max". Видит 4 фото, описание, характеристики (Бренд: Apple, Память: 256 ГБ). Есть 3 варианта: Black (в наличии, 129 990 руб.), White (нет в наличии), Blue (в наличии, 129 990 руб.). Маша выбирает Black, видит "В наличии: 5 шт.", нажимает "В корзину".

### Что нужно спроектировать

- Как получить полную информацию о товаре?
- Нужен ли отдельный запрос для получения SKU, или они приходят вместе с товаром?
- Как отображать выбор SKU, если у товара много вариаций?
- Нужен ли отдельный эндпоинт "проверить, в избранном ли товар"?

### Что если...?

> Покупатель открыл карточку товара, видит "В наличии: 1 шт.". Пока он читал описание (3 минуты), другой покупатель купил последнюю штуку. Маша нажимает "В корзину" -- что произойдет? В какой момент проверять наличие?

## Домен 3: Корзина

### Что делает

Покупатель собирает товары, которые хочет купить. Может добавлять, удалять, менять количество. Видит итоговую сумму. Из корзины переходит к оформлению заказа.

### Какие данные нужны

- Список позиций корзины (sku_id, quantity) -- из своей БД
- Данные о каждом SKU (название, фото, цена, наличие) -- из B2B
- Данные о товаре для каждого SKU (название товара) -- из B2B

### User story

> Маша добавила 2 товара в корзину: iPhone (1 шт.) и чехол (2 шт.). Открывает корзину. Видит:
> - iPhone 15 Pro Max 256GB Black -- 1 шт. -- 129 990 руб.
> - Чехол силиконовый iPhone -- 2 шт. -- 2 990 руб. (за штуку)
> - Итого: 135 970 руб.
>
> Маша меняет количество чехлов на 1. Итого пересчитывается: 132 980 руб. Нажимает "Оформить заказ".

### Что нужно спроектировать

- Добавление SKU в корзину (с проверкой: что если SKU уже в корзине?)
- Удаление позиции из корзины
- Изменение количества
- Получение корзины (с подгрузкой данных из B2B)
- Очистка корзины целиком
- Как считать итоговую сумму?

### Что если...?

> Покупатель добавил товар в корзину вчера. Сегодня открывает корзину -- а товар заблокирован модератором (или удален продавцом). Как корзина должна себя вести? Показать ошибку? Молча удалить? Показать "Товар недоступен"?

## Домен 4: Оформление заказа

### Что делает

Покупатель нажимает "Оформить" в корзине. Система проверяет наличие, резервирует товар в B2B, создает заказ, очищает корзину. Покупатель может смотреть свои заказы и их статусы.

### Какие данные нужны

- Позиции корзины -- из своей БД
- Актуальные данные SKU (цена, наличие) -- из B2B
- Результат резервирования -- из B2B (POST /api/v1/reserve)
- Заказ и его позиции -- в свою БД

### User story

> Маша нажимает "Оформить заказ". Система:
> 1. Берет все позиции из корзины Маши
> 2. Запрашивает актуальные данные по каждому SKU из B2B
> 3. Проверяет: все SKU доступны? activeQuantity >= запрошенного количества?
> 4. Отправляет POST /api/v1/reserve в B2B
> 5. B2B резервирует (activeQuantity -= quantity)
> 6. Создает заказ в своей БД с зафиксированными ценами
> 7. Очищает корзину Маши
> 8. Маша видит "Заказ #501 оформлен!"

> Через час Маша заходит в "Мои заказы", видит:
> - Заказ #501 -- 132 980 руб. -- Статус: Оплачен
> Нажимает -- видит детали: что заказала, по какой цене.

### Что нужно спроектировать

- Создание заказа из корзины
- Получение списка заказов пользователя (с пагинацией)
- Получение деталей одного заказа
- Отмена заказа (с возвратом резерва)
- **Машина состояний заказа** — для каждого перехода укажите:
  - Кто инициирует переход? (покупатель через API? Администратор через Django admin? Система автоматически?)
  - При каких условиях переход разрешён?
  - При каких запрещён?
  - Какие побочные эффекты (события, изменения в B2B)?

Известные статусы: `CREATED → PAID → ASSEMBLING → DELIVERING → DELIVERED`. Есть ещё `CANCELLED`.

Для отмены заказа: на каком статусе покупатель ещё может отменить, а на каком уже нет? Что происходит с резервом в B2B при отмене?

### Что если...?

> Покупатель оформляет заказ из 3 позиций. Резервирование в B2B прошло успешно. Но в этот момент упала БД B2C, и заказ не записался. Товар зарезервирован, но заказа нет. Что делать? Как откатить резерв? (Подсказка: это называется "компенсирующая транзакция".)

> Покупатель оформил заказ, B2B подтвердил резерв. Статус стал `ASSEMBLING` — склад уже начал сборку. Покупатель хочет отменить. Что вернуть покупателю? Что сделать с резервом? Нужен ли статус `REFUNDED` — или `CANCELLED` покрывает всё?

## Домен 5: Избранное

### Что делает

Покупатель отмечает понравившиеся товары. Может вернуться к ним позже. Избранное привязано к товару (product), а не к SKU -- потому что покупатель "лайкает" товар целиком, а конкретную вариацию выберет при покупке.

### Какие данные нужны

- Список product_id в избранном -- из своей БД
- Данные о товарах (название, фото, цена, наличие) -- из B2B (батч-запрос)

### User story

> Маша листает каталог, видит интересный рюкзак -- нажимает "сердечко". Потом видит кроссовки -- тоже добавляет. Через неделю заходит в "Избранное":
> - Рюкзак Xiaomi Mi Casual -- 3 490 руб. -- В наличии
> - Кроссовки Nike Air Max -- ТОВАР НЕДОСТУПЕН
>
> Рюкзак доступен -- Маша может перейти к карточке и купить. Кроссовки заблокированы модератором -- Маша может только удалить из избранного.

### Что нужно спроектировать

- Добавление товара в избранное
- Удаление товара из избранного
- Получение списка избранного (с подгрузкой данных из B2B)
- Как отображать товары, которые стали недоступны?

### Что если...?

> У покупателя в избранном 50 товаров. Чтобы показать страницу, нужно запросить данные для всех 50 из B2B. Это 50 HTTP-запросов? Или один батч? Что быстрее? Что если B2B тормозит -- ждать все 50 или показать что есть?

## Домен 6: Главная страница (баннеры + подборки)

### Что делает

Покупатель заходит на NeoMarket и видит главную: баннеры (слайдер с промо-акциями), подборки товаров ("Хиты продаж", "Новинки сезона"), дерево категорий. Баннеры и подборки создаются маркетологами через Django admin.

### Какие данные нужны

- Баннеры (image, title, link) -- из своей БД (создаются через Django admin)
- Подборки (title, product_ids) -- из своей БД (создаются через Django admin)
- Данные товаров для подборок -- из B2B (батч-запрос по product_ids)
- Дерево категорий -- из B2B

### User story

> Маша заходит на neomarket.ru. Вверху -- слайдер с баннером "Скидки на электронику до -30%". Ниже -- подборка "Хиты продаж" (8 карточек товаров). Еще ниже -- "Новинки" (8 карточек). Слева -- дерево категорий. Маша нажимает на баннер и попадает в категорию "Электроника" с примененным фильтром.

### Что нужно спроектировать

- Как получить список активных баннеров?
- Как получить список подборок с товарами?
- Нужен ли один эндпоинт "дай все для главной" или несколько отдельных?
- Как ограничить количество товаров в подборке при отображении?

### Что если...?

> Маркетолог создал подборку из 20 товаров. 5 из них заблокированы модератором. B2C запрашивает 20 товаров батчом из B2B, получает 15 валидных. Показать 15? Или дозапросить еще, чтобы показать ровно 20?

## Домен 7: Рекомендации

### Что делает

На карточке товара внизу -- блок "Похожие товары". Простая логика: рандомные товары из той же категории. Не машинное обучение, не коллаборативная фильтрация -- просто случайная выборка.

### Какие данные нужны

- Категория текущего товара -- из B2B (уже есть в карточке)
- Рандомные товары из той же категории -- из B2B
- Если товаров мало -- из родительской категории

### User story

> Маша смотрит карточку iPhone 15 Pro Max (категория: Смартфоны). Внизу видит "Похожие товары": Samsung Galaxy S24, Google Pixel 8, Xiaomi 14 -- все из категории "Смартфоны", выбраны случайно. Если бы в "Смартфонах" было всего 2 товара -- система показала бы товары из "Электроники" (родительская категория).

### Что нужно спроектировать

- Отдельный эндпоинт для рекомендаций или часть ответа карточки?
- Сколько товаров показывать (4? 8?)?
- Как исключить текущий товар из выборки?
- Как определить, что в категории мало товаров, и перейти к родительской?

### Что если...?

> Товар находится в самой глубокой категории "Apple > iPhone > iPhone 15" и в ней он один. Родительская "iPhone" -- 3 товара. "Apple" -- 10 товаров. До какого уровня подниматься, чтобы набрать 4 рекомендации?

## Формат ответа

Когда будете проектировать эндпоинты, для каждого укажите:

```
МЕТОД /путь

Описание: что делает

Параметры запроса:
  - param1 (тип) -- описание

Тело запроса (если POST/PUT):
  { json }

Ответ (успех):
  { json }

Ответ (ошибка):
  { json }

Коды ответов:
  200 -- OK
  400 -- Bad Request (что именно может быть неверно)
  404 -- Not Found (что не найдено)
```
